# 慢寻路分析系统

## 功能概述

系统现在会自动记录每5秒内最慢的寻路操作，并智能分析导致性能下降的原因。

## 数据记录

### 记录内容
每条慢寻路记录包含：
- **时间戳**: 记录时的系统运行时间(ms)
- **障碍物ID**: 发生慢寻路的障碍物编号
- **耗时**: 寻路计算耗时(ms)
- **起点/终点**: 寻路的起点和终点坐标
- **直线距离**: 起点到终点的欧几里得距离
- **障碍物数量**: 附近参与计算的障碍物数量
- **节点数量**: 可视图节点总数(估算)
- **路径长度**: 最终路径的节点数
- **成功状态**: 是否找到可行路径
- **分析结果**: 自动生成的慢速原因分析

### 存储限制
- 最多保存 **20条** 最近的慢寻路记录
- 超过限制时自动删除最旧的记录

## 性能分析逻辑

系统从以下5个维度分析慢寻路原因：

### 1. 距离因素
```
> 30000 单位  → "超长距离"
> 20000 单位  → "长距离"
```

### 2. 障碍物密度
```
障碍物密度 = 障碍物数量 / (距离 / 1000)

密度 > 15    → "障碍物极度密集"
密度 > 8     → "障碍物密集"
数量 > 100   → "障碍物过多"
```

### 3. 可视图复杂度
```
节点数 > 2000 → "可视图节点过多"
节点数 > 1000 → "可视图节点较多"
```

### 4. 路径复杂度
```
寻路失败      → "寻路失败-无可行路径"
路径长度 > 50 → "路径极其曲折"
路径长度 > 30 → "路径曲折"
```

### 5. 性能阈值
```
耗时 > 20ms  → "严重性能问题"
耗时 > 10ms  → "明显性能瓶颈"
```

## 优化建议

根据分析结果采取相应措施：

### 1. "超长距离" / "长距离"
- **问题**: 起终点距离过远导致计算量大
- **优化**: 
  - 限制移动障碍物的最大随机目标距离
  - 实施分段寻路机制

### 2. "障碍物极度密集" / "障碍物密集"
- **问题**: 局部区域障碍物过于密集
- **优化**:
  - 减少静止障碍物数量(当前150个可考虑再减少)
  - 调整障碍物生成算法，避免聚集

### 3. "可视图节点过多"
- **问题**: 节点数量导致A*搜索空间爆炸
- **优化**:
  - 已实现：动态距离阈值剪枝
  - 已实现：可见性缓存
  - 可选：实现分层导航网格

### 4. "路径极其曲折"
- **问题**: 迷宫状地形导致路径节点过多
- **优化**:
  - 已实现：路径平滑算法
  - 检查地图生成逻辑，避免产生迷宫

### 5. "寻路失败"
- **问题**: 目标点被完全包围或在障碍物内部
- **优化**:
  - 已实现：adjustTargetIfBlocked 调整目标点
  - 已实现：pushStartOutside 调整起点

## 典型问题示例

### 案例1: 地图边缘寻路
```json
{
  "distance": 42318.5,
  "obstacleCount": 45,
  "timeMs": 28.34,
  "analysis": "原因: 超长距离(>30000), 严重性能问题"
}
```
**原因**: 对角线跨越整个地图
**解决**: 限制目标生成范围

### 案例2: 狭窄通道
```json
{
  "distance": 8432.2,
  "obstacleCount": 124,
  "nodeCount": 498,
  "timeMs": 15.67,
  "analysis": "原因: 障碍物极度密集, 明显性能瓶颈"
}
```
**原因**: 两排障碍物之间的狭窄通道
**解决**: 调整障碍物尺寸和间距

### 案例3: 死胡同
```json
{
  "success": false,
  "obstacleCount": 78,
  "timeMs": 22.45,
  "analysis": "原因: 寻路失败-无可行路径, 严重性能问题"
}
```
**原因**: 目标点被包围
**解决**: 增强目标验证逻辑

## 实时监控

系统会：
1. 每5秒自动分析并打印最慢寻路
2. 持续积累慢寻路记录供后续分析
3. 通过API可随时查询历史记录

## 性能基准

**健康指标**:
- 平均寻路时间: < 2ms ✅
- 最大寻路时间: < 10ms ✅
- 可视图节点数: < 800 ✅

**警告阈值**:
- 平均寻路时间: 2-5ms ⚠️
- 最大寻路时间: 10-20ms ⚠️
- 可视图节点数: 800-1500 ⚠️

**严重问题**:
- 平均寻路时间: > 5ms ❌
- 最大寻路时间: > 20ms ❌
- 可视图节点数: > 1500 ❌
